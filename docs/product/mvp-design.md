# 自由研究ナビ MVP / v1.x 現行仕様

この文書は、自由研究ナビの MVP / v1.x 系における現行仕様の正本である。  
Issue の議論や過去文書よりも、現時点の実装・今後の改良判断ではこの文書を優先する。

## 1. MVP の目的

MVP で検証したいことは次の二点である。

1. 利用者の願いが、今日やれる一歩に変わるか
2. 利用者が使ったあとに「次にこれを試してみよう」と言えるか

特に重要なのは、**問いが動かない時期、記録しかできない時期でもループが切れずに続くか**である。

## 2. 現在の基本ループ

1. 願いを言葉にする
2. 任意で、今いちばん気になることを書く
3. AI が問い候補を整える
4. 利用者が今育てる問いを1つ選ぶ
5. その願いを見る観測骨格の中で、今回注目する記録項目を整える
6. 記録する
7. 振り返る
8. 次の一歩を決める

## 3. 問い生成の仕様

### 3.1 入力

問い生成に使う基本入力は次の通り。

- 何を願っているか
- なぜそう願うのか
- 今できていること
- まだできていないこと
- できるようになりたいこと
- 今いちばん気になること（任意）

### 3.2 生成方針

- AI は問いをゼロから作るより、本人の問いの芽を整えることを優先する
- 本人の入力がある場合、その言葉から離れすぎない
- 候補は一回で試せて、観察・記録可能なものにする
- 候補は最大3件を基本にする

### 3.3 候補の出し方

候補には、少なくとも次の違いを持たせる。

- そのまま進める
- 小さくする
- くらべやすくする

## 4. 記録項目設計の仕様

### 4.1 問い選択後のステップ

問いを選んだあとに、その願いにひもづく観測骨格のうち、今回の問いで注目する記録項目を整えるステップを持つ。

同じ願いのもとでは、観測したい世界そのものは比較的安定しているとみなす。  
そのため、記録項目は毎回ゼロから問い専用に作るのではなく、既存項目の再利用を基本にしながら、必要なら追加や細分化を行う。

利用者は、

- AI が提案した初期項目をそのまま使ってもよい
- 項目を追加・解除してもよい
- 必要なら自分で項目を追加してもよい
- active な問いの途中でも、問いを作り直さずに項目選択やラベルを見直してよい

### 4.2 項目候補の役割

記録項目候補は次の役割に分けて提示する。

- まず残す
- 違いを見る
- 気になったら足す

### 4.3 項目の説明

各記録項目は、少なくとも次を持つ。

- 項目名
- 入力型
- 何を見るための項目か
- あとでどう使うか

### 4.4 項目の継承

同じ願いの下で問いが変わっても、既存の観測項目はできるだけ引き継ぐ。  
MVP では、少なくとも次を支える。

- 既存項目の再利用
- 新規項目の追加
- 既存項目をより細かく分けるための細分化
- 問いごとの `使う / 使わない` の保持

### 4.5 手動追加

利用者は、自分で記録項目を1件ずつ追加できる。  
これは題材固有の観点を反映するためであり、AI 提案だけに固定しない。

### 4.6 観測値優先

記録項目は、集計値や評価値ではなく、観測値を優先する。

避けるものの例:

- 勝率
- 成功率
- 平均
- 合計
- スコア
- 評価
- 割合
- 差

必要な場合も、1回ごとに直接記録できる元の観測値に分解する。

### 4.7 active な問いでの後編集

active な問いでは、問いを作り直さなくても記録項目を後から編集できる。

- 今回注目する項目の追加・解除
- ラベルや説明の見直し
- optional 項目の採用
- 手動追加項目の追加・修正

既存記録との整合性は次の通りとする。

- 保存済みの記録本文と値は書き換えない
- 既存記録は、同じ key の項目としてあとから更新されたラベル・説明で見返す
- 問いごとの `使う / 使わない` は更新するが、願いの観測骨格から項目自体は消さない
- ただし、一度も採用されていない未使用候補は、後編集や問い確定の時点で観測骨格に残さない

### 4.8 細分化の品質

「細分化する」提案は、単なる選択肢追加ではなく、観測の解像度が上がるものだけを採用する。

- 悪い細分化: 親項目の言い換えや選択肢追加にとどまる
- 良い細分化: 何の違いを見るかが増え、あとで比較や解釈に使える

細分化候補には、少なくとも次を持たせる。

- なぜその細分化が有効か
- あとでどう比べるか

## 5. 記録と振り返り

### 5.1 記録

- active な問いに対して記録を積み上げる
- 記録には時間情報を基本項目として必ず持たせる
- MVP v1.2 では、時間情報の入力方式は `日時` を基本とする
- 記録は一覧として残り、あとで見返せる
- 記録一覧は新しい時間順に扱える必要がある
- 記録一覧の並び順は `recordedAt` を基準にする
- 記録項目は願いにひもづく観測骨格の一部として蓄積され、今回の問いの見方に対応している必要がある
- 問いで使わなかった項目も、願いの観測骨格からは消さずに残す

### 5.2 振り返り

振り返りでは少なくとも次を整理する。

- 今日気づいたこと
- まだ気になること
- 次にやってみたいこと
- 進み方の手ごたえ

## 6. MVP で重視する UX

- 完璧な問いを最初に決めさせない
- 記録しかできない時期を前進として意味づける
- AI に全部決めさせず、本人が少し選べるようにする
- 願いと観測骨格と問いの関係が見えるようにする
- 問い候補や記録項目候補が出たあと、利用者が次に見る場所を見失わないように導く

## 7. 正本の扱い

- 現行仕様の判断はこの文書を正本とする
- 個別 Issue は意思決定ログとして残す
- 実装に影響する重要判断は、Issue だけでなくこの文書に反映する

## 8. 現行画面構成

### 8.1 ホーム

ホーム画面では、現在の問いと最近の進み方に加えて、同じ願いの下で積み上がる観測のまとまりが感じられるようにする。

- 今の願い
- 今の問い
- 状態ラベル
- 最近の記録件数
- ここまでのようす
- 今見ている項目
- よく見る項目
- 今はお休みしている項目
- 見方の地図への導線
- 次の一歩
- キャラクターのひとこと

MVP v1.2 では、キャラクター画像は固定のデフォルト表現を維持する。  
ユーザー写真ベース表現は将来検討とし、初版では安全性・家族運用・保存方針の整理を優先する。

### 8.1.1 観測骨格の構造画面

観測骨格の構造画面では、active な願いにひもづく観測項目の親子関係と使用状況を見られるようにする。

- 親項目と細分化項目のツリー
- 今の問いで使っている項目
- 各項目が何回使われたか
- 最近増えた項目
- 今はお休みしている項目

### 8.2 問い作成

問い作成画面は、次の段階で構成する。

1. どこから始めるか
2. 願いの今を整理する
3. 問い候補を出す
4. 問いを1つ選ぶ
5. 観測項目を整える
6. この問いで始める

問い作成画面の観測項目調整では、少なくとも次を分けて見せる。

- 今回注目する項目
- 今回は使わないが骨格には残っている項目
- 細かく分ける候補

また、問い作成画面では次を満たす。

- 問い候補生成後に、候補が出た位置を見つけやすいこと
- 候補カードの選択状態が、カード全体で分かること
- 観測項目の役割である `まず残す / 違いを見る / 気になったら足す` の意味が見て分かること

### 8.3 記録

記録画面では、active な問いに対して記録を積み上げる。

- 問い文
- 目的ラベル
- 時間情報
- 今回注目する観測項目一覧
- active な問いの記録項目を後から見直す導線
- メモ
- 画像追加
- 記録一覧

### 8.4 振り返り

振り返り画面では、その日の記録を受けて次を整理する。

- 今日気づいたこと
- まだ気になること
- 次にやってみたいこと
- 進み方の手ごたえ

## 9. 現行 API 構成

### 9.1 問い関連

- `POST /api/ai/question-candidates`
  - 願い入力と `next_curiosity_text` をもとに問い候補を返す
- `POST /api/questions`
  - 選んだ問いと、その問いで注目する観測項目設定を保存する
- `GET /api/questions/active`
  - 現在の active question を返す

### 9.2 記録項目関連

- `POST /api/ai/record-fields/suggest`
  - 願いにひもづく既存観測項目と、今回の問い向けの追加候補を返す
- `PATCH /api/questions/active/field-definitions`
  - active な問いの記録項目定義と選択状態を更新する

### 9.3 記録関連

- `GET /api/records`
  - 記録一覧を返す
- `POST /api/records`
  - 記録を保存する
- `PATCH /api/records/:id`
  - 記録を更新する
- `DELETE /api/records/:id`
  - 記録を削除する
- `POST /api/records/:id/attachments`
  - 画像を追加する

### 9.4 振り返り関連

- `GET /api/reflections/today`
- `POST /api/reflections`
- `PUT /api/reflections/:id`

## 10. 現行データ構造の要点

### 10.1 Wish

- 願いの単位
- 願いそのものと、その背景・現在地を持つ
- 願いごとの比較的安定した観測骨格を持つ

### 10.2 Question

- 今育てる小さな問い
- active は常に 1 本
- `purposeFocus` を持つ
- 観測骨格のどこに注目するかを持つ

### 10.3 ObservationFieldDefinition

願いに紐づく観測項目定義として、少なくとも次を持つ。

- `key`
- `label`
- `type`
- `options`
- `role`
- `why`
- `howToUse`
- `isDefault`
- 必要なら、どの項目から細分化されたかを示す関係

### 10.4 QuestionObservationFocus

問いごとの焦点設定として、少なくとも次を持つ。

- どの観測項目を今回使うか
- その問いでは使わない項目も含めた選択状態
- 並び順

### 10.5 Record

- 問いに紐づく観測・試行の記録
- `kvFields` に構造化項目を保持する
- `memo` と画像添付も持てる

### 10.6 Reflection

- 日次の振り返り
- 気づいたこと、まだ気になること、次にやってみたいこと、進み方の手ごたえを持つ

## 11. 現行 UX 制約

- 問い候補は多すぎないようにする
- 記録項目候補は役割別に分ける
- 記録項目は意味が分かるようにする
- 本人が選び直せる余地を残す
- 手動追加項目は 1 件ずつ軽く足せることを優先する
- AI に全部設計させて終わりにしない
- 同じ願いの下では、既存観測項目をできるだけ再利用できるようにする
