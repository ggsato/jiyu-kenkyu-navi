# 採用済み判断

この文書は、Issue ベースで採用済みになった重要判断を短くまとめた一覧である。  
長い議論の本文は各 Issue に残しつつ、採用事項だけをここで追えるようにする。

## Issue #1

- 問い・記録・振り返りのループを回し続けられることを重視する
- 家族内の軽量ユーザー切り替えを持つ MVP とする

## Issue #2

- AI は問いの作者ではなく、本人の問いの芽を整える役に寄せる
- `next_curiosity_text` のような任意入力を持つ
- 問い選択後に「記録項目を整える」ステップを持つ
- 記録項目は役割つき候補 + 本人選択にする
- 本人が手動で記録項目を追加できる
- 記録項目は観測値を優先し、集計値を直接持ち込まない
- 記録項目には「何を見るためか」と「あとでどう使うか」を持たせる

## Issue #3

- Issue は意思決定ログとして残す
- 現行仕様の正本は文書に再統合する
- 壊してはいけないことは短い別文書で明文化する
- AI に渡す最小コンテキストと文書更新ルールを定義する

## Issue #5

- 記録項目設計の重心を、問い依存より願い依存へ寄せる
- 同じ願いのもとでは、比較的安定した観測骨格を持つ
- 小さな問いは、その観測骨格のどこを見るかを決める焦点設定として扱う
- 問い選択後の「記録項目を整える」ステップは維持する
- ただし毎回ゼロから専用項目を作るのではなく、既存項目の再利用・追加・細分化を行う
- AI は毎回新規項目を設計する役より、長く役立つ観測軸と今回注目する項目を提案する役に寄せる
- 問いごとの `使う / 使わない` は履歴として保持し、未選択項目も願いの観測骨格には残す
- MVP では、項目関係のうち継承と細分化を扱う

## Issue #9

- 記録の時間情報は任意ではなく基本項目とする
- MVP v1.2 の入力方式は `日時` を基本とする
- 記録一覧と振り返りは時間順に扱えることを前提にする
- 記録一覧の表示順は保存時刻ではなく `recordedAt` を基準にする

## Issue #10

- MVP v1.2 ではキャラクター画像は固定のデフォルト表現を維持する
- 写真ベース表現は将来検討とし、安全性・家族運用・保存方針の整理を先に行う

## Issue #11

- 細分化は、選択肢追加ではなく観測の解像度を上げる提案だけを採用する
- 細分化候補には `why` と `howToUse` を持たせ、何が見えるようになるかを説明する

## Issue #12

- active な問いでは、問いを作り直さずに記録項目を後から編集できる
- 保存済み記録は書き換えず、同じ key を通じて見方だけを更新する
- 一度も採用されていない未使用候補は観測骨格に残さない

## Issue #13

- 継続提案では `unit` `type` `options` `howToUse` を既存項目メタデータとして参照する
- 再利用優先度は直前の問いだけでなく、長期の利用回数と継続履歴も考慮する

## Issue #17

- 問い候補や記録項目候補を出したあと、利用者が次に見る場所を見失わない導線を持つ
- 問い候補カードと観測項目カードは、選択状態がカード全体で分かる見た目にする
- 観測項目の役割 `まず残す / 違いを見る / 気になったら足す` は説明つきで見せる
- 問い作成画面には、現在地へ戻れる簡易ステップナビを持たせる
- 願い入力欄には、短い意味説明や記入例を添えて迷いを減らす
- 記録削除は即時実行せず、記録一覧上で確認導線を挟む
- `Reflection` は `Wish` の土台を自動上書きしない
- `Reflection` はホームの声掛け、次の一歩、次の問い作りの参照材料として使う

## Issue #23

- UI 方針を「全部見せる」ではなく「段階的に見せる」に統一する
- 初回表示では最小限の情報だけ見せる
- 詳細情報は、必要になったときに開ける構成を優先する
- AI の初期提案を主表示で優先し、利用者の主操作は「選ぶ」「少し直す」に寄せる
- ホームは情報一覧より、「今どこにいて次に何をするか」が先に分かる構成を優先する

## Issue #30

- ホーム上部の主役は 1 つに絞り、`今の願い → 今の問い → いまの段階 → 次の一歩 → 主ボタン` を一本の導線として見せる
- ホームの初回表示では、主操作を 1 つ、補助操作を 1 つまでに抑えて迷いを減らす
- `今の願いの次の問い` や `別の願いを始める` は二段目の導線として扱う
- 右カラムは主役ではなく補足とし、ひとことや最近の記録など軽く読める情報へ寄せる
- ホーム内の「次」系表現は `次の一歩` を基準に整理する
- `いまの段階` と `最近の進み方` は、別情報ではなく 1 つの `今の状態` として判断が伝わる見せ方を優先する
- 問い作成系の導線は問い作成ページ側にも存在するため、ホームでは常設せず冗長さを避ける
- `だれが使うか` はホーム本文の主役ではなく、アプリ全体の利用者コンテキストとしてヘッダー側へ寄せる

## Issue #32

- 現行構造は壊さず、`Wish.observationFields` を「その願いで使う試し方・見方・結果の残し方の棚」として読み替える
- `QuestionObservationFocus` は「今回の問いで使う試し方 / 見方のセット」として扱う
- `Record.kvFields` は「今回何を試し、どういう状況で、どうだったか」の構造化記録として扱う
- 第一段では schema / API を固定変更せず、UI 文言・AI 指示・記録画面・ホーム表示の主語を置き換える
- 問いの利用者向け分類は `見る / 試す / 続ける` を優先し、既存の `compare / relate / predict` は内部値として当面維持する
- `試す` の問いでは、`compare` role の項目を1つだけ主軸として扱い、その項目は原則 `select` + 2件以上の選択肢で明示する
- `試す` の問いでは、比較して見る対象も Question ごとに 1 つ明示し、可視化と AI 要約はその項目を優先して使う
- 記録画面では、`やり方 -> 状況 -> 結果` の順で理解しやすい並びを優先する
- ホームや構造画面では、「観測項目」より「試し方 / 見方の棚」が先に伝わる見せ方を優先する
- 最小可視化では、日単位中心を原則とし、`見る` は願いに近い項目の系列、`試す` は試し方ごとの日系列比較、`続ける` は続けている試し方が読めるときの系列を優先する
- この日単位の圧縮要約は、次の問い生成と記録項目生成の AI 入力にも渡してよい

## Issue #34

- 記録ページは、生データの入力・修正・構造編集に責務を絞る
- 記録ページの主役は、その問いで使っている全項目を並べた表にする
- レコードの追加・編集は、表を見たまま使える非遮断型ポップアップで行う
- 項目編集は常設カードではなく、表ヘッダーから開く補助操作へ寄せる
- 記録ページでは圧縮や次の判断を主責務にしない
- 圧縮・比較・次の判断材料づくりは `流れを見る` 画面へ分離する
- `流れを見る` で整理した内容は、次の問い作成の参照メモとして使う
- 願いの土台 (`Wish.current_state / not_yet / desired_state`) は `流れを見る` で自動上書きしない
- 問い作成での自動反映は、まず `next_curiosity_text` への補助利用に留める
- AI に渡す主入力も、生記録の羅列より `流れを見る` の圧縮要約と整理メモを優先する
